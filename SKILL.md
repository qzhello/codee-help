# Code Reading Skill

当用户希望先建立项目整体认知，再按问题逐步深挖代码和知识点，并沉淀为可复用文档时，使用这个 skill。

这个 skill 分为五种命令语义：

- `codee-init`：只做项目初始化扫描，建立宽视角架构地图，不做全面精读
- `codee-help`：统一问答入口。围绕用户当前问题补读代码、知识点、命令或文档，并按需要转调其他命令
- `codee-web`：把现有阅读产出整理为适合浏览的 HTML 手册
- `codee-impact`：当用户准备修改代码时，沿调用链和依赖关系评估影响范围，并给出修改建议与验证范围
- `codee-command`：整理运维命令、管理接口、脚本入口与参数说明，维护单独的命令手册

目标不是一次性把整个仓库读完，而是形成一套可持续演进的项目理解产出，帮助：

- 人快速建立对项目的整体认知
- AI 在后续任务中快速接手并定位上下文
- 将零散阅读过程沉淀为结构化知识

## 核心原则

- 先建地图，再钻细节；先确认事实，再做推断。
- 只基于代码、配置、脚本、文档、git 历史等可验证证据下结论。
- 区分“已确认”“高概率推断”“未知待验证”。
- 输出必须可复用，不能只是一次性的阅读笔记。
- 重点先行，拒绝废话。
- 默认用技术人的视角看问题，不做情绪化表达，不做无效道歉。
- 能用表格、清单、调用链、图示表达的，就不要堆长段文字。
- 能用一两句话说清楚的，不要拉长成大段解释。

## 安装与调用

### Codex

推荐安装位置：

- `~/.codex/skills/codee/`

安装方式：

1. 创建目录 `~/.codex/skills/codee`
2. 把当前目录下的全部文件复制进去
3. 确保至少包含：
   - `SKILL.md`
   - `agents/openai.yaml`
   - `references/`
   - `examples/`
   - `evals/`
   - `scripts/`

调用方式：

- 直接在问题里提到 `codee`
- 或显式说：
  - `use codee-init`
  - `use codee-help`
  - `use codee-impact`
  - `use codee-command`
  - `use codee-web`

如果宿主支持自定义 slash 命令，也可以把统一入口绑定为 `/codee`。

### Claude Code

Claude Code 没有一个完全统一的 skill 目录约定，不同工作流常见两种接入方式：

1. 作为项目内命令说明接入
2. 作为全局或项目级 prompt/command 模板接入

推荐做法：

- 在项目内放一份 `codee` 命令说明，例如 `.claude/commands/codee.md`
- 说明 `/codee` 是统一入口，再在说明中约定：
  - 需要全局架构时走 `codee-init`
  - 需要回答具体问题时走 `codee-help`
  - 需要评估改动时走 `codee-impact`
  - 需要整理命令时走 `codee-command`
  - 需要生成 HTML 手册时走 `codee-web`

如果你的 Claude Code 环境支持自定义 prompts/skills，只需要把本目录内容按其要求复制进去，并保留这里的命令语义即可。

### 统一入口建议

建议统一从 `/codee` 入口提问，例如：

- `/codee 帮我找一下 seaweedfs 的压缩实现在哪里`
- `/codee 我想要运维手册中的 volume.list 命令的参数`
- `/codee 我准备改 FUSE flush 逻辑，先帮我分析影响范围`

## 建议目录结构

推荐将这个 skill 组织为：

- `SKILL.md`：主说明，定义工作流、标准和输出要求
- `agents/openai.yaml`：UI 展示信息
- `references/`：阅读产出模板、检查清单、术语表等参考资料
- `examples/`：高质量输入输出示例，帮助稳定风格与深度
- `evals/`：用于评估这个 skill 是否真正产出高质量阅读结果
- `assets/`：可复用的非上下文资产，如模板文件、示意图骨架、导图底板

安装与调用细节见：

- `references/install-and-usage.md`

其中：

- `examples/` 适合放“给定仓库后，应该产出什么样的阅读结果”的示例
- `evals/` 适合放“如何判断这次阅读是否合格”的任务与标准
- `assets/` 当然可以写上，而且很有价值，只是它更适合放会被复用或复制的资源，而不是说明文字

## 命令定义

### codee-init

`codee-init` 的目标是“扫架构，不深挖”。

这一阶段只需要建立项目的宽视角认知：

- 项目使用的语言、框架、运行时、构建工具
- 仓库目录结构与顶层模块边界
- 核心入口、启动方式、部署方式、测试入口
- 主要依赖、外部系统、工具链、版本信息
- 粗粒度架构分层和主要功能域

`codee-init` 不应该做的事：

- 不要试图从头到尾精读所有核心流程
- 不要对每个功能点展开调用链细读
- 不要替用户预判所有后续问题并提前展开

`codee-init` 的产出重点是：

- 架构总览
- 功能域导航
- 关键词映射表
- 官方仓库地址与官方文档入口
- 后续可继续提问和补读的入口

### codee-help

`codee-help` 的目标是“用户问什么，就帮他找到答案，并根据问题类型选择合适的阅读或分析方式”。

适用于：

- 用户对某个模块、功能、接口、调用链有具体问题
- 用户忘了之前某个知识点，需要重新建立上下文
- 用户要把某个技术点沉淀为专题手册，方便以后快速介入

`codee-help` 必须：

- 围绕用户问题限定范围
- 只补读当前问题相关的文件、模块、调用链、git 改动
- 输出一份可以快速复习的专题文档
- 维护关键词映射表，支持后续按关键字快速跳转
- 如存在官方文档、官方仓库、官方设计说明，优先记录入口
- 适当怀疑代码本身；如果源码存在明显缺陷、危险实现、设计漏洞或严重可维护性问题，应直接指出
- 必要时根据用户问题转调：
  - `codee-init`：当用户缺少整体上下文
  - `codee-impact`：当用户准备改代码
  - `codee-command`：当用户查询命令、接口、脚本
  - `codee-web`：当用户要求生成 HTML 手册

### codee-web

`codee-web` 的目标是把已有 Markdown 阅读产出转成 HTML 手册。

HTML 产出应满足：

- 首页可导航
- 支持从大功能到子功能的层级浏览
- 明显展示版本、最新修改时间、产出日志入口
- 可快速跳转到关键词索引和专题页

### codee-impact

`codee-impact` 的目标是“用户想改代码时，快速分析影响范围”。

适用于：

- 用户要修改某个函数、方法、类、接口或模块
- 用户想知道某处改动会影响哪些调用方和下游行为
- 用户需要快速评估测试范围、回归风险和相关 git 演进

`codee-impact` 必须输出：

- 目标修改点
- 直接调用链
- 反向调用方
- 关联文件与关联模块
- 可能受影响的接口、配置、测试
- 高/中/低风险分级
- 建议验证清单

### codee-command

`codee-command` 的目标是“把运维命令、管理接口、脚本入口整理成可维护手册”。

适用于：

- 项目存在大量 CLI 命令、admin API、shell 脚本、运维工具
- 用户希望维护一份命令参数、示例、用途、风险说明的阅读手册
- 需要单独生成命令手册 HTML 页面

`codee-command` 必须输出：

- 命令或接口名称
- 功能说明
- 参数说明
- 示例
- 风险提示
- 相关文件
- 相关功能域

## 官方资料规则

如果项目存在明确的官方资料，文档中应尽量记录：

- 官方仓库地址
- 官方文档地址
- 官方设计文档、RFC、Wiki、API 文档地址

其中“官方仓库地址”在初始化文档中应尽量作为固定字段保留。

如果仓库内已有明显线索，例如：

- `README`
- `go.mod`
- `package.json`
- 仓库 git remote
- 项目官网链接

应优先从这些位置提取官方入口，而不是凭印象填写。

## 官方文档补读规则

遇到以下情况时，可以并且应当补读官方文档，再回写总结：

- 当前问题拿不准
- 问题风险较高，错误理解会导致错误修改或错误决策
- 用户已经多轮提问，但仅靠本地代码仍无法稳定解释清楚
- 本地实现与标准协议、官方行为、公共接口规范强相关

补读官方文档时要遵守：

- 优先查“官方”来源，而不是零散博客
- 结论要区分“代码中观察到的行为”和“官方文档规定的行为”
- 最终把关键知识总结回当前文档，避免下一次重复查找

## 回答风格

每次回答默认遵守：

- 先给结论，再给证据，再给建议动作
- 不说套话，不说寒暄，不说无意义的安抚
- 不写“抱歉”“可能不太准确但”“我觉得”等低信息密度前缀
- 优先使用：
  - 表格
  - 调用链
  - 路径清单
  - 风险分级
  - 对比项
- 除非用户明确要细节，否则默认简洁

推荐回答顺序：

1. 重点结论
2. 关键证据
3. 下一步建议

如果某个问题适合图表表达，优先输出：

- 模块表
- 参数表
- 调用链
- 影响范围矩阵
- 命令速查表

## 文档生成边界

默认不要因为“分析过了”就自动生成文档。

只有在以下情况下才生成或更新正式文档：

- 用户明确要求生成文档、手册、总结、HTML 页面
- 当前任务属于运维命令手册或接口手册整理
- 需要把多轮结论沉淀为长期复用材料，且用户已经表达出该意图

如果只是一次性问答、定位、分析、review、影响评估，默认直接回答，不要额外落文档。

简单原则：

- 有必要才生成
- 用户没提就不要自动写

## Bug 修复规则

如果用户的问题本质上是“定位并修复 bug”，默认流程不是立刻改代码，而是：

1. 先定位问题
2. 推演调用链、状态变化和触发条件
3. 给出明确结论：
   - 根因是什么
   - 为什么会发生
   - 修复思路是什么
   - 可能影响哪些模块和测试
4. 先向用户确认是否执行修改

在用户明确确认之前，不要擅自修改代码。

如果只是阅读、诊断、review、定位问题，可以继续分析；但一旦进入“要不要改”的分界点，必须先给方案，再等用户确认。

## 语言环境确认规则

如果任务依赖具体语言或运行时版本，进入实现、编译、运行、测试之前，应先确认当前会话环境是否与用户预期一致。

重点包括但不限于：

- Java / JDK
- Go
- Node.js
- Python
- Rust
- Maven / Gradle / pnpm / npm / yarn

默认流程：

1. 先识别项目声明的版本
   - 例如 `go.mod`、`pom.xml`、`.java-version`、`package.json`、`.nvmrc`、`pyproject.toml`
2. 再核对当前会话可用版本
3. 如果版本可能影响结果，先向用户确认：
   - 当前会话版本是否就是目标版本
   - 是否允许按当前环境继续
4. 用户确认后，再进入下一步实现或验证

如果只是做只读分析，可以先继续；但只要涉及：

- 编译
- 运行
- 测试
- 依赖安装
- 生成物验证

就必须先确认环境版本，避免在错误环境下给出误导性结论。

## 阅读目标

整体能力至少覆盖以下内容：

- 项目使用的语言、框架、运行时、构建工具
- 仓库目录结构与模块边界
- 架构分层、核心组件、关键职责
- 依赖关系与外部集成
- 关键调用链与主流程
- 配置、启动入口、部署或运行方式
- 测试结构与质量信号
- 近期 git 改动、热点模块、演进方向
- 适合后续 AI 介入的上下文入口与任务切分点

## 标准工作流

按命令类型区分工作流，避免把初始化和精读混在一起。

### codee-init 工作流

#### 1. 建立项目地图

先读取能快速建立全局认知的内容：

- 根目录文件和一级子目录
- `README`、`Makefile`、`package.json`、`pyproject.toml`、`go.mod`、`pom.xml`、`Cargo.toml`、`Dockerfile`、CI 配置等
- 主入口文件、启动脚本、构建脚本、部署脚本

这一阶段要回答：

- 这是一个什么项目
- 主要语言和技术栈是什么
- 如何启动、构建、测试
- 目录大致如何分层

#### 2. 识别架构骨架

围绕目录、模块、命名和依赖，识别：

- 哪些是入口层、接口层、领域层、数据层、基础设施层
- 哪些目录是真正的核心业务
- 哪些目录是封装、适配、工具、共享库
- 模块之间大致如何依赖

如果项目规模较大，优先画出“模块关系图”的文字版，不急于覆盖每个文件。

#### 3. 产出初始化导航文档

`codee-init` 结束时，只产出宽视角文档，不展开所有调用链。

至少包含：

- 项目概览
- 技术栈与版本
- 顶层目录与模块职责
- 粗粒度架构图
- 功能域列表
- 关键词映射表
- 推荐下一步可深挖的问题

### codee-help 工作流

#### 1. 锁定问题范围

先明确当前问题属于哪一类：

- 某个功能
- 某个模块
- 某条调用链
- 某个技术机制
- 某段 git 演进
- 某个命令、接口或运维问题
- 某个需要生成手册或 HTML 的整理任务

然后只读取和这个问题直接相关的代码与文档。

如果用户的问题本质上是：

- “先帮我建立全局上下文” -> 优先按 `codee-init` 处理
- “我要改这里，帮我看影响” -> 优先按 `codee-impact` 处理
- “这个命令/接口怎么用” -> 优先按 `codee-command` 处理
- “把这些手册变成 HTML” -> 优先按 `codee-web` 处理
- “这是个 bug，帮我修” -> 先按 `codee-help`/`codee-impact` 完成定位和方案说明，再征求是否修改

#### 2. 找出核心运行路径

围绕最重要的用户动作或系统主流程，梳理调用链：

- 请求从哪里进入
- 参数如何传递和转换
- 核心业务逻辑在哪里执行
- 数据从哪里读取、写入或发送
- 错误处理、鉴权、缓存、消息队列、异步任务在哪里介入

调用链要尽量写到函数、类、文件、模块级别，不要只停留在概念层。

#### 2.1 审视源码质量

在 `codee-help` 阶段，不要默认相信当前实现一定合理。

如果发现以下情况，应明确指出：

- 调用链不一致或语义冲突
- 错误处理缺失或吞错
- 配置与实际行为不一致
- 接口命名、参数、返回值存在误导
- 并发、权限、数据一致性、资源释放、边界条件存在严重问题

如果问题严重，应单独给出“严重问题”小节，而不是藏在普通备注里。

#### 3. 识别依赖与外部集成

明确：

- 一方依赖和三方依赖
- 数据库、中间件、对象存储、RPC、HTTP API、消息系统等外部系统
- 配置项如何影响行为
- 环境变量、密钥、开关、feature flag 在哪里生效

如果依赖很多，只记录对主流程和系统演进最关键的部分。

#### 4. 结合 git 理解演进

阅读 git 时重点关注：

- 最近一段时间的主要改动方向
- 高频改动文件和模块
- 大的重构、迁移、接口变更
- 哪些区域在活跃演进，哪些区域相对稳定

不要把 git 日志简单堆出来，要提炼“项目正在往哪里演化”。

#### 5. 沉淀专题产出

最终输出必须形成一份体系化文档。默认参考 `references/reading-output-template.md`。

如果用户没有指定格式，优先输出一份适合 AI 接手的结构化专题文档。

### codee-web 工作流

#### 1. 读取现有 Markdown 产出

优先基于已存在的初始化文档、专题文档、关键词索引生成 HTML。

#### 2. 组织导航层级

HTML 默认按以下层级组织：

- 项目
- 大功能域
- 子功能
- 具体专题

#### 3. 输出 HTML 手册

确保首页能快速引导阅读，而不是把所有内容铺平成一页长文。

### codee-impact 工作流

#### 1. 锁定目标改动点

先确认用户准备修改的是：

- 哪个文件
- 哪个符号
- 哪个功能域

如果用户描述不完整，先根据文件名、关键词或方法名做定位。

#### 2. 建立影响范围

至少分析：

- 当前符号直接调用了谁
- 谁在调用当前符号
- 所属包/模块依赖关系
- 是否实现了某个接口，或被某个接口约束
- 是否受配置、特性开关、协议字段影响

#### 3. 补充验证范围

进一步收集：

- 相关测试
- 近期相关 git 改动
- 相邻热点文件
- 可能需要联动修改的文档或脚本

#### 4. 输出 impact 文档

impact 结果必须偏行动导向，而不是只画调用图。

### codee-command 工作流

#### 1. 识别命令与接口入口

优先扫描：

- CLI 子命令注册
- 管理接口路由
- shell 脚本
- 运维脚本
- 内部工具命令

#### 2. 结构化整理

每个命令或接口至少记录：

- 名称
- 用途
- 参数
- 示例
- 风险提示
- 代码入口

#### 3. 生成命令手册

命令手册可以同时维护：

- Markdown 版索引
- HTML 版界面

## 阅读方法

### 证据优先

优先使用以下证据来源：

- 代码入口和调用关系
- 构建与运行配置
- 测试代码
- 类型定义、接口定义、协议文件
- git 历史
- 官方仓库和官方文档

如果某个判断缺少证据，不要写成确定事实。

### 从粗到细

推荐节奏：

1. 先 `codee-init` 做目录、入口、模块与边界
2. 再用 `codee-help` 深挖某个主流程、知识点、命令或调用链
3. 如果用户准备改代码，再用 `codee-impact` 评估影响范围
4. 如果项目存在较多运维命令或管理接口，再用 `codee-command` 沉淀命令手册
5. 补充依赖、配置、测试、git 演进
6. 形成增量文档

### 对大仓库的处理

面对大型项目时：

- 先锁定一个主应用或主服务
- 再识别与其直接相关的核心模块
- 把非核心目录标为“暂不深入”
- 不要为了“全覆盖”牺牲结构化理解

## 输出要求

### codee-init 输出要求

初始化文档至少要包含：

- 项目概览
- 官方仓库地址
- 官方文档入口
- 技术栈
- 版本与工具链
- 目录结构与模块职责
- 架构理解
- 主要依赖与外部系统
- 配置与运行方式
- 测试与质量观察
- 功能域导航
- 关键词映射表
- 推荐下一步阅读方向

### codee-help 输出要求

专题文档至少要包含：

- 问题定义
- 范围边界
- 关键调用链
- 依赖与外部系统
- 关键文件与关键方法
- 相关官方资料
- 必要的配置、数据结构或协议
- 相关 git 演进
- 风险、疑点、待验证项
- AI 快速介入指南

其中必须额外包含“关键词映射表”。

### codee-impact 输出要求

impact 文档至少要包含：

- 目标改动点
- 所属功能域
- 关键调用链
- 反向调用者
- 关联文件与模块
- 潜在影响面
- 建议修改范围
- 建议验证范围
- 相关测试与 git 改动
- 风险等级

### codee-command 输出要求

命令手册至少要包含：

- 命令/接口名称
- 类型：CLI / HTTP API / Shell / Internal Tool
- 功能说明
- 参数表
- 示例
- 风险提示
- 相关文件
- 相关功能域
- HTML 页面入口

### 关键词映射表要求

关键词映射表必须帮助读者避免全文扫读。

至少包含：

- 关键词
- 所属功能域
- 一句话说明
- 关键文件
- 关键方法
- 推荐先读顺序

### AI 快速介入指南要求

“AI 快速介入指南” 必须明确给出：

- 从哪里开始读最有效
- 哪几个文件是最高价值入口
- 哪些模块最值得优先建立上下文
- 后续提问时应该提供哪些关键词或文件路径

## 文档组织规则

产出文档必须按“功能从大到小”严格分类，而不是按时间顺序平铺。

例如：

- `seaweedfs/`
- `seaweedfs/s3/`
- `seaweedfs/s3/put.md`
- `seaweedfs/s3/get.md`
- `seaweedfs/s3/delete.md`
- `seaweedfs/s3/multipart.md`
- `seaweedfs/fuse/`
- `seaweedfs/fuse/read.md`
- `seaweedfs/fuse/write-flush.md`

如果某个大功能下还有子功能，继续按层级下钻。

不要把多个不相干专题塞进一个长文档里。

impact 文档也应按功能树存放，例如：

- `seaweedfs/s3/impact-put.md`
- `seaweedfs/fuse/impact-write-flush.md`

命令手册可单独存放，例如：

- `seaweedfs/commands/index.md`
- `seaweedfs/commands/s3-shell.md`
- `seaweedfs/commands/admin-api.md`

## 版本与日志要求

主文档必须显式标记：

- 文档版本
- 对应仓库版本或提交号
- 最新修改时间

每次产出或更新后，都要把记录追加到产出日志中，而不是覆盖历史。

产出日志至少记录：

- 时间
- 动作类型：`codee-init` / `codee-help` / `codee-web` / `codee-impact` / `codee-command`
- 本次新增或更新的专题
- 关联仓库版本

## 脚本设计建议

推荐用 Python3 做 orchestration，用语言原生工具做分析内核。

建议脚本结构：

- `scripts/codee_init.py`
- `scripts/codee_help.py`
- `scripts/codee_web.py`
- `scripts/codee_impact.py`
- `scripts/codee_command.py`
- `scripts/codee_command_web.py`
- `scripts/scanners/repo_fingerprint.py`
- `scripts/scanners/keyword_index.py`
- `scripts/scanners/tree_sitter_index.py`
- `scripts/scanners/git_hotspots.py`
- `scripts/scanners/go_pkg_graph.py`
- `scripts/scanners/go_call_graph.py`

### 通用分析建议

- 用 `tree-sitter` 或 `ast-grep` 做结构化扫描
- 用 `rg` 做高速文本初筛
- 用 git 历史补充热点模块和近期演进

### Go 项目建议

如果目标仓库是 Go，优先使用 Go 原生工具辅助：

- 包依赖图：`godepgraph`
- 调用图：`golang.org/x/tools/cmd/callgraph`
- 需要更细能力时，可用 `golang.org/x/tools/go/callgraph`

### impact 分析原则

影响范围分析不是只找“谁调用了它”，还要看：

- 这个符号是否被接口约束
- 是否有导出行为变化
- 是否涉及序列化字段、HTTP/RPC 协议、数据库结构、配置项
- 是否影响测试、脚本、部署或运维流程

## 表达规范

- 用中文输出。
- 多用短句和结构化标题，减少空泛描述。
- 结论后尽量附上证据路径或文件位置。
- 对推断内容加明确标记，例如“推断”“待验证”。
- 如果仓库中存在不一致或历史包袱，要明确指出，不要强行美化成整洁架构。

## 质量标准

一份合格的阅读产出应满足：

- 新读者可在短时间内理解项目主干
- 后续 AI 能据此快速定位上下文
- 结论与代码证据能互相对应
- 对架构、依赖、调用链、演进四类信息都有覆盖
- 能明确指出未知项，而不是假装理解完整

## 产出模板

默认读取：

- `references/reading-output-template.md`
- `examples/example-output.md`
- `evals/eval-checklist.md`

当用户要求更偏某一方向时再加权：

- 偏架构：加强模块边界、分层、依赖方向
- 偏排障：加强入口、调用链、配置、日志、错误处理
- 偏二开：加强扩展点、插件机制、接口与数据模型
- 偏接手维护：加强 git 演进、热点文件、测试覆盖和风险点

## 示例与评估

### examples

在以下场景优先读取 `examples/`：

- 用户要求“给个样例产出”
- 需要对齐 `codee-init` 的宽视角文档
- 需要对齐 `codee-help` 的专题文档
- 需要判断“体系化阅读产出”到底应该写到什么程度

示例的价值在于：

- 让输出不只停留在泛泛总结
- 让 AI 知道每一节应该写哪些事实
- 帮助后续复用同一套阅读格式

### evals

在完成阅读产出后，用 `evals/` 做一次自检。至少确认：

- `codee-init` 是否只做了宽扫，没有提前深读过多细节
- 是否覆盖语言、架构、依赖、调用链、git 演进
- 是否给出了明确证据路径
- 是否区分了事实、推断、待验证项
- 是否提供了 AI 快速介入指南
- 是否提供了关键词映射表
- 是否能帮助一个陌生读者在短时间内接手项目

如果某次输出无法通过这些检查，应继续补充阅读，而不是直接结束。

### assets

`assets/` 适合放这类内容：

- 文档模板底稿，如空白 Markdown 模板、表格模板
- 架构图底板、Mermaid 片段、流程图骨架
- 可复制到目标仓库中的产出文件骨架

不要把长篇说明写进 `assets/`。说明应放在 `SKILL.md` 或 `references/`。
